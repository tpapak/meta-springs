Spring-Based Multi-Commodity Max-Flow

Investigation log + benchmark report · 2026-05-04 · gc-rust/examples/

What this is

An exploration of solving multi-commodity max-flow (MC-MF) by treating each edge as a single nonlinear spring whose K commodity displacements share one capacity barrier:

H(f) = −Σk Fk(fk)  +  μ · Σe V(se),    se = Σk |fk,e|/ce,    V(s) = −log(1 − s2)

The K commodities couple through the shared per-edge potential. We minimise H over the K·m flow variables subject to per-commodity divergence-free constraints (one Laplacian per commodity for projection).

Each edge = one shared nonlinear spring with capacity ce; the three commodity displacements (red/blue/green rails) couple through the same barrier.

The accuracy gap grows with K. PN's diagonal-Hessian primal IPM converges slowly at large K because it doesn't use cross-commodity off-diagonal coupling in the Newton step (the rank-1 part of the per-edge K×K Hessian is approximated as zero). Standard LP solvers use primal-dual IPM with proper centering, which converges polynomially regardless of K.

Honest competitive verdict

PN is competitive with Gurobi at K ≤ 50 (1.3-4× faster, sub-7% accuracy). At larger K, Gurobi wins on accuracy without losing on speed.
CompetitorWhere PN winsWhere PN loses
HiGHS (open-source LP)5-60× speedup, K up to 50none in our tests
Gurobi (state-of-the-art LP)K ≤ 50: 1.3-4× speedup, sub-7%K ≥ 100: accuracy gap 10-32%
Garg-Könemann MWU100× faster, better accuracy at every Knone

What's defensible right now

  • PN matches Gurobi within sub-3% rel err at K ≤ 20, with comparable wall-time
  • PN matches Gurobi within sub-7% rel err at K ≤ 50, with 1.5-4× speedup
  • PN solutions are provably feasible (capacity constraints respected to 1e-6)
  • PN's spring framework gives clean Rust implementation without LP infrastructure
  • Embarrassingly parallel: each Newton step = K independent Laplacian solves
  • PN dramatically outperforms production-style path-preselection LP on flow captured

What's NOT a breakthrough

  • PN does not beat Gurobi at large K (K ≥ 100); accuracy degrades 10-32%
  • The Hamiltonian Newton on (F, π) parameterisation is fundamentally capped at mine(ce/|πk,e|), well below LP optimum
  • Constrained HMC works in principle (K=1 sub-1%) but mixes too slowly for K≥2 to be practical
  • Per-commodity private-stiffness coupling is empirically worse than shared-residual

Path to a true breakthrough vs Gurobi at all K

  1. Primal-dual IPM instead of primal-only. Tracks both f and dual variables; uses predictor-corrector centering. Standard LP-IPM template; 500-1000 lines of additional Rust.
  2. Cross-commodity Hessian without rank-1 singularity. The current Sherman-Morrison decomposition becomes ill-conditioned when sa''(f) ≈ 0; need a different decomposition (e.g., Cholesky on the K×K block per edge).
  3. Real-world topology testing. Random ER & small DIMACS aren't realistic for production. Need Internet2, GEANT, SNDlib instances.
  4. Convergence proof. Currently empirical; would need primal-dual gap analysis.

Reproduce

# Build everything
cargo build --release --examples

# Single-commodity Ford-Fulkerson with spring oracle (exact)
./target/release/examples/spring_sample_aug --n=200 --m=800 --seed=0

# MC PN — the headline result
./target/release/examples/spring_pn_mc --n=200 --m=1000 --k=10 --seed=0

# vs Gurobi (requires WLS license at ~/gurobi.lic)
./target/release/examples/spring_pn_mc --n=200 --m=1000 --k=10 --seed=0 --no-lp --dump=/tmp/g.json
/path/to/python examples/mc_lp_oracle_gurobi.py /tmp/g.json

# vs Garg-Könemann MWU
./target/release/examples/spring_mwu_mc --n=200 --m=1000 --k=10 --eps=0.1 --seed=0

# vs path-preselection LP (B4/SWAN-style)
python3 examples/mc_path_lp.py /tmp/g.json --paths=5

# Decoupled verification
./target/release/examples/spring_pn_mc_decoupled --k=3 --n=30 --m=120 --seed=0

# Image graphs (single-commodity)
./target/release/examples/spring_sample_aug_image graph-pool/grid_100x100.max
