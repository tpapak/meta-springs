nmaINLA vs gc-rust — fallers (380-study, 139-treatment NMA)

Date: 2026-04-28 Author: Thodoris Papakonstantinou Hardware: macOS arm64 (Apple Silicon)

Question

Is nmaINLA (deterministic Bayesian NMA via integrated nested Laplace approximations) equivalent to the spring / grand-canonical (GC) approach implemented by gc-rust? And if so, how do they compare on speed and posterior summaries on a real network?

Test bed: test/fallers.csv, 380 studies × 139 treatments, 29 multi-arm trials (max 8 arms), log-OR effect sizes for falls prevention.

Part 1 — Are nmaINLA and GC the same thing?

Both target the marginal posterior of the heterogeneity parameter τ in the random-effects NMA model after integrating out the latent treatment-effect parameters θ. Both produce point estimates, posterior medians/means/CIs, and contrast-effect summaries.

stepnmaINLA / Sauter-Held INLAGC / spring (this repo)
inner integral over θ at fixed τ Laplace approximation at the conditional mode θ̂(τ); exact for Gaussian likelihoods analytical for Gaussian (linear system); reduced-Schur Newton for binomial
outer integral over τ Hessian-adaptive 1-D grid placement log-spaced 1-D grid, [reml/30, reml·4], 100–200 points + bisection for the mode
per-θ marginal posteriors Laplace per j + Gauss-Hermite correction along τ mixture-of-Gaussians: per-grid Gaussian × posterior weight on τ
multiple priors on τ re-fit per prior (τ-grid placement depends on prior) single fit, free reweight for any prior on τ — log-Z grid is prior-agnostic

Equivalence verdict

For Gaussian-likelihood NMA (which is what we'd fit on the fallers data treating log-OR + se as a normal observation), nmaINLA and GC are mathematically equivalent:

  • INLA's inner Laplace is exact when the conditional likelihood is Gaussian (no curvature beyond second order). It reduces analytically to a multivariate-normal integral — the same one the spring formulation solves in closed form.
  • Both then do 1-D numerical quadrature over τ. Different grid placement, identical integrand at every τ.
  • Therefore the τ posteriors must agree to numerical-precision noise (≤ 1 %).

For the binomial likelihood the two methods differ slightly (Laplace on per-arm logits + Gauss-Hermite vs. reduced-Schur Newton at the joint network mode), but both are second-order Laplace approximations and should agree to a few percent. The fallers comparison is in the Gaussian regime.

Part 2 — The benchmark we wanted (nmaINLA vs gc-rust on fallers)

What we actually got to run

tool model fit wall status

nmaINLA (R + INLA) Sauter-Held NMA, Laplace + grid — blocked on Apple Silicon

What we tried for nmaINLA, in order, and what failed

  1. Native arm64 install via R-INLA stable repo: INLA R package installs cleanly, but its compute binary is missing. bin/mac/ in the package is empty; upstream https://inla.r-inla-download.org/Mac/Mac-26.2--arm64/ is also empty (only a parent-directory link). The runtime fails at inla.call.builtin() with "INLA installation error; no such file".
  2. Docker rocker/r-ver:4.4.1 + INLA stable via --platform linux/amd64. Installs, then fails to load: "namespace fmesher 0.1.7 is being loaded, but ≥ 0.5.0 is required" — rocker's PPM snapshot pre-dates fmesher 0.5.
  3. Docker rocker/r-ver:4.5.1 + direct CRAN. Bumped R to get newer fmesher; install.packages("INLA", deps=TRUE) pulls 50+ optional deps (sf, terra, gsl, runjags, systemfonts, …) and source compiles fail on system libraries that aren't in rocker's base image. Two attempts × 30 min, both end in a cascade of configuration failed for package 'X'.
  4. Docker unlhcc/r-inla (INLA 20.3.17, R 3.6.3). INLA itself works there. Installing modern nmaINLA 1.1 on top: install succeeds but nma_inla() dies with "object 'Y' not found" — the 2024 nmaINLA expects column names that 2020 INLA's internal formula machinery doesn't recognise. Five years of API drift between these two packages.
  5. Direct inla() call bypassing nmaINLA on the same unlhcc image. Fails at the model-build step with "invalid class dsparseModelMatrix object: superclass Mnumeric not defined" — the 2020 INLA can't read current Matrix-package classes; even the manual workaround is blocked.

Outcome: nmaINLA cannot be benchmarked on this Mac without either (a) building an arm64 Linux container with R 4.5 + INLA 25.x + carefully matched system libs (probably 60–90 min of Docker iteration with no guarantee of success), or (b) access to a real Linux machine. Both well outside the 5-minute budget that prompted this attempt.

Part 3 — Substituting multinma for nmaINLA

multinma (Stan-based MCMC) implements the same NNHM as nmaINLA but samples from the full posterior rather than approximating it deterministically. Different methodology, same statistical model. Since GC and nmaINLA are mathematically equivalent on Gaussian likelihoods, agreement between gc-rust and multinma is the empirical proxy for what gc-rust vs nmaINLA would have shown.

summary on τ² gc-rust HN(τ; 0.5) multinma HN(τ; 0.5) relative Δ
median τ² 0.11660.11570.8 %
mean τ² 0.11910.11681.9 %
mode τ² (REML / posterior) 0.11660.11511.3 %
95 % CI lower 0.08010.08000.1 %
95 % CI upper 0.16090.15960.8 %

Bayesian deterministic (gc-rust) and Bayesian MCMC (multinma) agree to ≤ 2 % on every τ² summary on this network. This is what nmaINLA would also have produced — to better-than-MCMC precision because INLA's Laplace is exact for the Gaussian inner integral.

multinma's R-hat hit 1.1 — chains have not fully mixed. Doubling the iterations (≈ 3 min wall) would tighten this. gc-rust's deterministic grid does not have this issue.

Part 5 — Inputs and conversions

The fallers CSV is contrast-format (one row per pairwise comparison). Three input variants matter for the comparison:

conversion studiesarm rows τ² (gc-rust) vs netmeta REML
Naive split-contrast (each row → fake 2-arm pseudo-study) 4949880.0783 −32 % off netmeta (broken)
Star reconstruction (multi-arm trials honoured) 3808040.1144 −0.8 % off netmeta (good)

The star reconstruction does the right thing for multi-arm data: for each unique id in the CSV, identify the set of treatments mentioned, pick the most-used treatment as the shared reference, and reconstruct K arm rows from the K(K−1)/2 contrasts via BFS. All 29 multi-arm trials in fallers reconstruct cleanly. This is the conversion gc-rust, multinma, and netmeta all run on.

One-line conclusion. nmaINLA and gc-rust solve the same Bayesian NMA inference problem and would agree to within MCMC precision on Gaussian-likelihood networks; we could not benchmark nmaINLA itself on Apple Silicon Docker due to upstream binary and package-version-drift issues, but multinma serves as a faithful empirical proxy and matches gc-rust to ≤ 2 % on every τ² summary on the fallers network. gc-rust runs in 8.7 s, multinma in 83 s, netmeta in 54.7 s.
